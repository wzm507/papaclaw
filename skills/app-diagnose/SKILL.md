# 应用排查

应用搭建过程中，将用户感知到的异常现象还原为可定位、可修复的具体问题。通过逐步引导确认页面、操作、预期与实际表现，避免在模糊描述下盲目修改代码。

## 铁律

**用户没说清楚问题之前，禁止改代码。**

用户说"提交之后好像不对"，这句话有五种可能——提交没反应、提交报错、提交成功但数据没出现、数据出现但内容不对、页面跳转错了。猜错一种就白改，甚至改出新问题。

## 什么时候触发

用户的描述符合以下特征时，启用本技能：

- 描述模糊，无法直接定位到代码：  "好像不对"、"有点问题"、"不好使"
- 只描述了现象，没说具体操作步骤："数据少了"、"页面有问题"
- 包含不确定性："好像"、"偶尔"、"之前还好的"

## 排查流程

### 第一步：定位页面

> "你说的问题出现在哪个页面？"

如果能读取应用的路由/导航结构，直接列出页面让用户选：
> "是以下哪个页面？
> 1. 首页
> 2. 订单列表
> 3. 提交申请
> 4. 其他（请描述）"

### 第二步：还原操作

> "你在这个页面上做了什么操作？比如点了哪个按钮、填了什么内容、选了哪个选项。"

如果用户描述不清，给具体选项：
> "你是：
> A. 打开页面就看到问题了（不需要任何操作）
> B. 填了表单点了提交
> C. 点了某个按钮或链接
> D. 在列表里做了筛选/搜索"

### 第三步：确认预期

> "你觉得正常的话，应该发生什么？"

### 第四步：确认实际表现

> "实际发生了什么？"

给出常见现象让用户选：
> "A. 页面白屏 / 什么都没显示
> B. 点了按钮没反应
> C. 出现了报错信息（如果有，请截图或复制）
> D. 操作看起来成功了，但数据/结果不对
> E. 页面跳到了错误的地方
> F. 其他（请描述）"

### 第五步：确认频率和时间

> "这个问题是每次都这样，还是偶尔出现？大概什么时候开始的？"

这一步可选——如果前四步已经足够定位问题，跳过。

### 第六步：复述确认

把收集到的信息组织成一句话，让用户确认：

> "我理解的问题是：在【订单列表】页面，点击【删除】按钮后，页面显示删除成功，但刷新后数据还在。每次都这样。是这个情况吗？"

用户确认后，再开始查代码。

## 引导原则

| 原则 | 说明 |
|------|------|
| 每次只问一个问题 | 不要一次丢出"在哪个页面、做了什么、看到什么" |
| 给选项优先 | 用户选比用户想更容易，也更准确 |
| 不要引导答案 | "是不是点了提交？"→ 错；"你做了什么操作？"→ 对 |
| 不要用技术语言 | "API 返回 500"→ 错；"服务器报错了"→ 对 |
| 不够就追问 | 信息不足以定位问题时，继续问，不要猜 |
| 够了就停 | 能定位问题了就别再问，用户会烦 |

## 复述模板

```
在【页面名称】页面，执行【具体操作】后，
预期：【应该发生什么】
实际：【实际发生了什么】
频率：【每次 / 偶尔 / 第一次遇到】
```

## 复述确认后：假设推演

问题明确后，不要直接改代码。先列出可能的原因，逐个排除。

### 原则：猜测不是排查

改代码之前必须有证据支撑。上来就改 = 猜，猜错了改出新问题。

### 第七步：列出 3-5 个假设

根据用户确认的问题，从四个方向思考可能的原因（互不重叠、完整覆盖）：

| 方向 | 考虑什么 | 示例 |
|------|---------|------|
| **数据** | 输入为空、字段缺失、类型不匹配、格式错误 | 表单提交时 `date` 字段传了字符串而非时间戳 |
| **逻辑** | 条件写反、漏了判断、顺序不对、边界情况 | 删除接口调了但前端没刷新列表 |
| **环境** | 配置项、权限、接口地址、部署状态 | 本地能用但线上不行，因为环境变量不同 |
| **状态** | 缓存过期、页面状态没重置、加载顺序 | 编辑后返回列表，显示的还是旧数据 |

每个假设写清楚：

```
假设 1（数据）：提交时 price 字段传了空值，后端校验报错但前端没显示错误信息
  支撑：用户说"点了提交没反应"——可能是请求发了但失败了
  矛盾：暂无
  验证：查看浏览器网络请求，看提交接口的响应状态码

假设 2（逻辑）：提交按钮的 onClick 事件没绑上
  支撑：用户说"点了没反应"——可能根本没触发
  矛盾：如果没绑上，应该连 loading 都不会有
  验证：读提交按钮所在组件的代码，看 onClick 是否绑定

假设 3（状态）：表单的 state 在重新打开弹窗时没重置，导致校验失败
  支撑：用户说"第一次能提交，后来就不行了"
  矛盾：暂无
  验证：检查 Modal 的 destroyOnClose 或 form.resetFields 调用
```

### 第八步：逐个验证

从最可能的假设开始验证：

1. **一次只验证一个假设**——不要同时改两处"看看哪个管用"
2. **验证手段尽量轻**——先读代码、看日志，不要上来就改
3. **每次验证最多改 5 行代码**——如果需要改更多，说明假设太模糊，拆细
4. **验证完记录结果**——确认了 / 排除了 / 还不确定

### 第九步：确认原因并修复

- 假设被证实 → 写修复代码，告诉用户改了什么
- 假设被排除 → 验证下一个
- 全部排除 → 回到第一步，补充观察，重新列假设

### 反推土机规则

以下情况出现时，**立刻停下来**：

| 信号 | 意味着什么 | 怎么做 |
|------|-----------|--------|
| 已经改了两次还没修好 | 方向可能就是错的 | 停下来，回到假设列表，看看是不是遗漏了什么 |
| 改完一个问题冒出另一个 | 可能在改症状而非根因 | 停下来，重新梳理数据流，从源头找 |
| 改动超过 20 行还"差一点" | 在一个错误方向上越陷越深 | 全部回退，重新列假设 |
| 有矛盾证据但选择忽略 | 假设有漏洞 | 正视矛盾，重新评估 |

## 修复后

告诉用户改了什么，让用户再试一次：

> "问题找到了：点击提交后表单数据发到了后端，但 price 字段传了空值导致保存失败，前端没有显示错误提示。
> 已修复：加了空值校验和错误提示。你再试一次看看。"

## 绝对不能做的事

| 错误 | 为什么错 |
|------|---------|
| 用户说"有问题"就开始翻代码 | 不知道问题是什么，翻了也白翻 |
| 根据模糊描述猜测原因直接改 | 猜错了改出新问题 |
| 一次问四五个问题 | 用户回答不过来，信息质量也差 |
| 追问太多轮 | 三四个问题还定位不了，说明问题可能需要打开浏览器实际看 |
| 用技术术语反问用户 | "是不是 CORS 问题？"——用户不知道这是什么 |
| 只列一个假设就开始改 | 那不叫排查，叫赌博 |
| 同时改两处"看哪个管用" | 改好了也不知道是哪个起的作用 |
| 改了两次没修好继续同方向改第三次 | 方向大概率就是错的，停下来重新想 |
