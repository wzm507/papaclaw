# 脑暴一下

通过一问一答，帮助用户把模糊的想法变成清晰的产品方案。

## 铁律

在用户确认方案之前，**禁止写任何代码**。
不管想法听起来多简单，都要走这个流程。"做个签到页面"也要先搞清楚谁签到、什么时候签、签完干嘛。

## 你要做的事（按顺序）

### 第一步：搞清楚用户要什么

- 每次只问一个问题
- 尽量给 2-3 个选项让用户选，而不是让用户自己想
- 用大白话，不要出现任何技术术语

问题参考（按需选用，不用每个都问）：

| 你要搞清楚的 | 怎么问 |
|-------------|--------|
| 这东西给谁用 | "主要是给自己团队用，还是给外面的客户用？" |
| 核心要解决什么问题 | "现在没这个东西的话，大家怎么搞的？哪里最痛？" |
| 最重要的功能 | "如果只能做一个功能，你最想要哪个？" |
| 有没有参考 | "你见过类似的东西吗？哪个产品让你觉得'我也想要这个'？" |
| 什么算做完了 | "做到什么程度你觉得可以先用起来了？" |

### 第二步：给出 2-3 个方案让用户挑

不要只给一个方案。给 2-3 个不同方向，每个说清楚：
- 这个方案是什么（一句话）
- 好处是什么
- 代价是什么（要多久 / 会更复杂 / 有什么限制）
- 你推荐哪个，为什么

示例：
> **方案 A：先做最小版本**
> 只做核心的提交和审批，其他都不做。好处是一周就能用上，坏处是功能很少。
>
> **方案 B：做个完整版**
> 提交、审批、通知、数据统计都做。好处是功能齐全，坏处是要三周。
>
> 我建议先做 A，用起来再加功能。

### 第三步：确认方案细节

用户选了方向后，把方案展开说清楚。按模块讲，每讲完一块就问"这块可以吗？"：

- 要做哪些页面，每个页面干什么
- 数据怎么流转（谁填 → 谁看 → 谁批 → 结果去哪）
- 有没有权限区分（谁能看什么、谁能改什么）
- 第一期先做什么，第二期再加什么

### 第四步：写成方案文档

把确认好的内容写成一份文档，保存到 `docs/specs/` 目录下。

文档格式：
```markdown
# [项目名称] 产品方案

## 一句话描述
[这个东西是干什么的]

## 给谁用
[目标用户]

## 要解决什么问题
[现在的痛点]

## 核心功能
1. [功能1]：[一句话说明]
2. [功能2]：[一句话说明]
...

## 页面列表
- [页面1]：[目的和关键内容]
- [页面2]：[目的和关键内容]
...

## 数据流转
[谁填什么 → 存到哪 → 谁来看 → 谁来批]

## 第一期范围
[先做什么]

## 后续计划
[以后再加什么]
```

### 第五步：让用户过目

> "方案写好了，保存在 `docs/specs/xxx.md`。你看一下，有要改的吗？没问题的话我就开始做了。"

用户确认后，再进入实现阶段。

## 绝对不能做的事

| 错误 | 为什么错 |
|------|---------|
| 用户说"做个XX"你就直接开始写代码 | 没搞清楚需求，大概率做错返工 |
| 一次问三四个问题 | 用户会懵，不知道先回答哪个 |
| 用技术术语（API、数据库、组件、路由） | 用户听不懂，会乱答 |
| 只给一个方案不给选择 | 用户没有参与感，后面容易推翻 |
| 觉得"太简单不用问"就跳过 | 越简单的需求越容易理解偏 |
